一起加入光荣的进化吧

作者：汪立坤 Email：wlk752@163.com

如果你目前使用 AI 辅助研究的方式仍然是：打开 ChatGPT 的对话框，粘贴问题，等待回复，再手动把结果复制回自己的文档——那么你正在使用的是一种已经过时的交互范式。

Chatbot 的交互逻辑本质上是你问我答，模型无法看到你的数据文件，无法运行分析脚本，无法直接修改你的论文草稿——研究者不得不充当翻译者，在对话框与工作环境之间反复搬运信息。当前，AI 应用正在经历从 Chatbot 到 Agent 的范式跃迁：新一代工具直接进入研究者的本地环境，自主地读写文件、执行代码、调用外部工具。在软件工程领域，这种以自然语言指挥 AI 工作的方式被称为 vibe coding；迁移到学术研究中，便是本文所讨论的 vibe researching。

本文档介绍依靠 Claude Code 进行 vibe researching 的基本方法，主要包括基础概念与安装、基本用法、技能系统（Skill）以及完整的研究工作流搭建。

1 Claude Code 基础

1.1 Claude Code 是什么

  1. Claude Code 是由 Anthropic 开发的终端 AI Agent 工具。在技术层面，Claude Code 本质上是一个"智能体安全框架"（Agentic Harness），它将 Claude 系列具备深度推理能力的模型与文件系统、Shell 环境、Git 版本控制工作流以及外部 API 协议（如 MCP）无缝集成。
  2. 通俗地讲，Claude Code 是一个运行在你电脑终端中的 AI 助手，你用自然语言告诉它要做什么，它就能直接在你的项目文件夹里帮你完成工作——读取你的数据文件、编写并运行分析脚本、整理文献笔记、修改论文草稿等等。
  3. 这种设计对社会科学研究者的关键意义在于：Claude Code 能够将你的整个项目文件夹作为工作上下文。它能同时看到你的数据文件、文献笔记、分析脚本和写作草稿，然后在这些真实材料上直接工作，而非像对话式 AI 那样在一个脱离你项目的对话框中泛泛而谈。这意味着你的角色从"亲自执行每一个操作步骤"转变为"研究的指挥者和质量把关人"——你负责提出问题、做出判断、把控方向，而 Claude Code 负责执行具体的技术操作。

1.2 为什么社会科学研究者最近才注意到 Claude Code 这类工具？

Claude Code 早在 2025 年便已发布，但直到 2026 年才逐渐进入社会科学研究者的视野。这一时间差并非偶然，而是由多重因素共同造成的。

  1. 首先是技术门槛的隔离。Claude Code 运行在终端（命令行）中，安装过程涉及 Node.js 环境配置、命令行操作和 API 密钥管理。这些步骤对软件工程师而言是日常操作，但社会科学研究者的日常工具栈——Word、SPSS、NVivo、Zotero——几乎全部依赖图形化界面，打开终端本身就已超出多数人的经验范围。
  2. 其次是信息传播的圈层隔离。Claude Code 最早的用户和传播者几乎全部来自软件开发社区，其发布渠道、教程和讨论集中于 GitHub、Hacker News 等开发者平台，这些平台与社会科学研究者的信息来源（学术会议、学科公众号、同行推荐）之间存在明显的信息壁垒。在 2025 年，社科领域讨论的主要话题仍停留在如何使用 ChatGPT 润色论文、如何借助 Kimi 阅读文献等层面。
  3. 第三是命名带来的认知偏差。Claude Code中的Code一词本身就会劝退大量非编程背景的研究者——我又不写代码，这跟我有什么关系？然而，这类工具的能力远不止于编写代码；只是其早期的宣传定位确实以开发者为核心受众，非编程场景下的应用价值尚未被充分揭示。

2026 年出现转折的催化因素包括：

  1. MCP（Model Context Protocol）协议的成熟使 Claude Code 得以连接 Zotero 等学术工具，降低了研究者对“这与我的工作流有何关联”的疑虑；
  2. 一批跨界研究者开始撰写中文教程并分享实操案例；
  3. 加之 vibe coding 概念的广泛传播，使非程序员群体开始意识到不会写代码也能使用这类工具。

1.3 人机协作的三个阶段

在介绍 Claude Code 之前，有必要先理解人与 AI 协作方式的演化脉络。理解这一背景，才能把握 Claude Code 这类工具的真正意义所在。

  1. 第一阶段：Prompt Engineering（提示词工程）：这是我们最熟悉的阶段。用户在 ChatGPT、Claude.ai、Kimi 等对话框中输入问题，模型返回一段文本，交互到此结束。这一阶段的核心技巧是"怎么问"——如何措辞才能获得更好的回答。其瓶颈在于，所有上下文都需要用户手动粘贴进对话框，模型的能力边界止于文本生成。用一个类比来说：这相当于通过电话咨询一位顾问，你说一句，对方答一句，但对方完全看不到你桌上的资料。
  2. 第二阶段：Context Engineering（上下文工程）：人们逐渐意识到，模型输出的质量不取决于一句提示词写得多巧妙，而取决于能否将正确的上下文提供给模型。于是出现了 RAG（检索增强生成）、系统提示词（System Prompt）的精细设计、长上下文窗口等技术。关注点从怎么问转向怎么让模型看到它需要看到的信息。MCP（Model Context Protocol）协议也属于这一阶段的产物——它解决的正是如何将 Zotero、数据库等外部工具中的信息接入模型上下文的问题。
  3. 第三阶段：Harness Engineering（智能体框架工程）：这是当前最新的阶段，也是 Claude Code 所处的位置。关注点不再只是给模型提供信息，而是设计一套完整的基础设施来约束和引导模型的自主行为——它能操作哪些工具、在什么条件下执行什么动作、如何处理错误、如何与版本控制系统交互、如何在多次会话之间保持记忆。
  4. 用一个类比来理解这三个阶段的区别：Prompt Engineering 像是给一个临时工下达一条口头指令；Context Engineering 像是给这个临时工提供一份详细的背景资料包；而 Harness Engineering 像是设计了一整套组织制度——岗位职责、工作流程、审核机制、档案管理——然后让 AI 在这套制度中运行。例如，Emory 大学经济学教授 Pedro Sant'Anna 为 Claude Code 设计的学术工作流便是一个典型案例：他构建了五层配置体系（项目宪法、领域规则、技能模块、专业审查员、跨会话记忆），其中批评者只读不改、修复者不能自我批准的对抗式审查机制，本质上就是一种制度设计。

本文档后续介绍的 Skill 系统和工作流搭建，正是属于第二和第三阶段的实践。理解这一点很重要：Claude Code 的价值不仅在于它是一个更强的 ChatGPT，而在于它提供了一个让研究者设计和实施 AI 治理结构的平台。

1.4 终端 AI Agent：Claude Code 与同类工具

理解了三个阶段之后，我们可以更准确地定位 Claude Code 所属的工具类别——终端 AI Agent（Terminal-based AI Agent）。这类工具直接运行在用户电脑的终端（命令行）中，能够自主地读写文件、执行代码、调用外部工具、管理项目结构。如果说 ChatGPT 是通过电话提供咨询的顾问，终端 AI Agent 就是直接坐在你电脑前、能够亲手替你操作的助手。目前这一赛道的主要工具包括：

工具 开发方 底层模型 特点
Claude Code Anthropic Claude 支持 MCP 协议，可连接 Zotero 等研究工具，生态最为完整
Codex CLI OpenAI GPT 思路相近，但协议生态不同
Gemini CLI Google Gemini Google 生态下的对应方案

2 Claude Code 的安装与配置

本节面向从未使用过终端的研究者。如果你已经熟悉命令行操作，可以直接跳至安装方法部分。

2.1 基础概念：在开始之前你需要了解的事

2.1.0 第零步：网络环境

Claude Code 的安装和使用全程需要访问 Anthropic 的海外服务器（claude.ai、api.anthropic.com 等）。对于中国大陆用户而言，这意味着你需要先解决稳定的国际网络访问环境，且确保终端（而非仅浏览器）的流量也经过该环境。这是一切后续操作的前提，请在开始安装前确认这一点。

2.1.1 什么是终端（Terminal）

终端，也叫命令行（Command Line）或控制台（Console），是你电脑上的另一种操作界面。你日常使用的是图形界面（GUI）——通过鼠标点击图标、拖拽文件来操作电脑。终端则是文本界面：你通过键入文字命令来告诉电脑做什么，电脑以文字形式返回结果。

这听起来像是倒退回了上世纪，但终端的优势在于精确和可自动化。Claude Code 就运行在终端中——你在终端里输入自然语言指令，Claude Code 读取后直接在你的文件夹中执行操作。

如何打开终端：

macOS：

  1. 按 Cmd + Space 打开聚焦搜索（Spotlight）
  2. 输入 Terminal（或"终端"）
  3. 按回车，终端窗口就会打开
  4. 你会看到一个带有闪烁光标的黑色（或白色）窗口——这就是终端

Windows：

  1. 按 Win 键，输入 PowerShell
  2. 点击"Windows PowerShell"打开
  3. 你会看到一个蓝色背景的文字窗口——这就是 PowerShell，Windows 上的终端

提示：在终端中，你输入一行命令后按回车（Enter）执行。如果不小心输入了错误的命令，通常可以按 Ctrl + C 中断。不用担心会搞坏电脑——Claude Code 在执行敏感操作前会征求你的确认。

2.1.2 什么是包管理器（Package Manager）

在图形界面中，你安装软件的方式通常是：打开浏览器 → 搜索软件 → 下载安装包 → 双击安装。包管理器是终端世界的"应用商店"，它让你用一行命令就能安装、更新或卸载软件。

你在安装 Claude Code 的过程中可能会遇到以下几个包管理器的名字：

  • Homebrew：macOS 上最流行的包管理器。如果你在 Mac 上使用终端，迟早会接触到它。
  • npm：Node.js 的包管理器。Claude Code 早期通过 npm 安装，目前已不再推荐这种方式，但你可能在旧教程中看到它。
  • WinGet：Windows 自带的包管理器。

好消息是：当前推荐的 Claude Code 安装方式（原生安装器）不依赖任何包管理器。 你只需要在终端中运行一行安装命令即可。包管理器是备选方案，初学者可以完全忽略。

2.1.3 什么是环境变量与 PATH

当你在终端中输入 claude 并按回车时，电脑需要知道"claude 这个程序在硬盘的哪个位置"。PATH 就是一个告诉电脑"去哪些文件夹里找程序"的列表。

安装 Claude Code 时，安装器会自动把程序所在的路径添加到你的 PATH 中。你通常不需要手动操作。但有一个关键细节：PATH 的更新需要重新打开终端才能生效。这就是为什么几乎所有安装教程都会提醒你"安装后关闭终端，重新打开一个新的终端窗口"。如果安装后输入 claude 提示"命令未找到"，十有八九是因为没有重启终端。

2.1.4 什么是 API Key 及其计费逻辑

API Key（应用程序接口密钥）是一串由字母和数字组成的字符串，相当于你使用 Claude Code 的通行证。它的作用是让 Anthropic 的服务器识别这个请求来自谁，并据此计费。

Claude Code 有两种付费模式：

  1. 订阅制（推荐初学者）： 购买 Claude Pro（$20/月）或 Claude Max（$100/月）计划后，直接用你的 Claude 账户登录即可使用 Claude Code，费用包含在月费中。这种方式最简单——不需要手动管理 API Key，登录即用。每 5 小时有使用额度上限，超出后需等待刷新。
  2. 按量付费（API 模式）： 在 Anthropic Console（console.anthropic.com）注册账户，充值后获取一个以 sk- 开头的 API Key。每次使用 Claude Code 时按实际消耗的 token（可理解为文字量）计费。这种方式适合偶尔使用、或需要精确控制成本的场景。

对社科研究者的建议：如果你刚开始尝试，直接购买 Claude Pro（$20/月）是最省心的方式。登录后就能用，不需要理解 token 计费这些技术细节。等你用熟了，再根据使用频率决定是否切换到 Max 或 API 模式。

2.2 安装方法

截至 2026 年，Anthropic 官方推荐使用原生安装器（Native Installer）。这是最简单的方式：无需预装任何依赖软件，安装后自动在后台更新。

2.2.1 macOS

打开终端，复制以下命令，粘贴到终端中，按回车执行：

curl -fsSL https://claude.ai/install.sh | bash

这行命令的含义是：使用 curl 工具从 Anthropic 的服务器下载安装脚本，然后用 bash 执行它。你不需要记住这些，复制粘贴即可。

系统要求：macOS 13.0（Ventura）或更高版本。Apple Silicon（M1/M2/M3/M4）和 Intel 芯片均支持。

2.2.2 Windows

Windows 上有两种方式：

方式一：原生安装（推荐）

  1. 首先安装 Git for Windows——这是一个图形化的安装包，下载后双击安装，全程点"Next"即可。Claude Code 在 Windows 上内部依赖 Git Bash 来执行命令，所以这一步不能跳过。

  2. 安装完 Git 后，打开 PowerShell（按 Win 键，输入 PowerShell，点击打开），复制粘贴以下命令并回车：

irm https://claude.ai/install.ps1 | iex

这行命令的含义：irm（Invoke-RestMethod）从 Anthropic 下载安装脚本，iex（Invoke-Expression）执行它。同样，复制粘贴即可。

如果 PowerShell 提示"无法运行脚本"（执行策略限制），先运行以下命令解除限制，然后重新执行安装命令：

Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

方式二：通过 WSL（Windows Subsystem for Linux）

WSL 会在你的 Windows 电脑上创建一个 Linux 虚拟环境。这种方式更接近 macOS/Linux 的使用体验，安全性也更好，但设置步骤稍多：

  1. 以管理员身份打开 PowerShell（右键点击 PowerShell → "以管理员身份运行"）
  2. 运行：
wsl --install -d Ubuntu
  1. 按提示重启电脑
  2. 重启后 Ubuntu 会自动打开，要求你创建用户名和密码（这是 Linux 环境的账户，与 Windows 登录密码无关）
  3. 在 Ubuntu 终端中运行：
curl -fsSL https://claude.ai/install.sh | bash

macOS vs Windows 怎么选？ 如果你同时有两台电脑，macOS 的安装体验明显更简单。如果只有 Windows，推荐方式一（原生安装），除非你之后打算深度使用 Linux 工具链，才需要考虑 WSL。

2.2.3 Linux

curl -fsSL https://claude.ai/install.sh | bash

支持 Ubuntu 20.04+、Debian 10+ 等主流发行版。

2.3 验证安装

安装完成后，关闭当前终端窗口，重新打开一个新的终端（回顾一下：这是为了让 PATH 更新生效）。然后运行：

claude --version

如果看到类似 2.x.x 的版本号输出，说明安装成功。你还可以运行健康检查命令，它会自动诊断常见问题：

claude doctor

2.4 首次登录与认证

在终端中运行：

claude

首次启动时，Claude Code 会自动打开你的默认浏览器，引导你登录 Anthropic 账户。按照页面提示完成登录后，返回终端，你会看到 Claude Code 的欢迎界面——认证就完成了。

登录凭据保存在本地 ~/.claude/ 目录中，后续启动无需重复登录。

2.5 首次使用：在项目文件夹中启动

认证完成后，用终端进入你的研究项目文件夹，然后启动 Claude Code：

cd ~/my-research-project
claude

cd 是什么？ cd 是 "change directory" 的缩写，意思是"切换到某个文件夹"。~/ 代表你的用户主目录（macOS 上是 /Users/你的用户名/，Windows 上是 C:\Users\你的用户名\）。例如，如果你的项目在桌面上的 "thesis" 文件夹中，命令就是 cd ~/Desktop/thesis。

此时 Claude Code 会自动扫描当前文件夹的内容作为工作上下文。你可以直接用自然语言告诉它你想做什么，例如：

  • "帮我看一下 data 文件夹里有哪些数据文件"
  • "读取 survey_data.csv，告诉我有多少个变量和观测值"
  • "帮我写一个 R 脚本，对 income 变量做描述性统计"

这就是 Claude Code 区别于对话式 AI 的核心体验：它直接工作在你的真实项目环境中，而不是一个脱离上下文的对话框。

2.6 常见问题

Q：安装后输入 claude 提示"命令未找到" A：关闭终端窗口，打开一个全新的终端，再试一次。如果仍然不行，运行 which claude（macOS/Linux）或 where claude（Windows）检查程序是否在 PATH 中。

Q：Windows 上 Claude Code 找不到 Git Bash A：在配置文件中手动指定路径。用文本编辑器打开（或创建）~/.claude/settings.json，写入：

{
  "env": {
    "CLAUDE_CODE_GIT_BASH_PATH": "C:\\Program Files\\Git\\bin\\bash.exe"
  }
}

Q：我应该选 Pro 还是 Max？ A：从 Pro（$20/月）开始。如果你发现自己经常遇到使用额度耗尽需要等待的情况，再考虑升级到 Max。

Q：Claude Code 是否有桌面应用？ A：有。如果你实在不想使用终端，Anthropic 提供了 Claude Code 的桌面应用（Desktop App），支持 macOS 和 Windows，可以在 claude.ai 下载。桌面应用提供图形化界面，功能与终端版本基本一致。但本文档后续介绍的进阶用法（如 Skill 系统、CLAUDE.md 配置）主要基于终端版本。

2.7.3 大陆用户镜像配置

在前文"第零步"中我们提到，大陆用户需要先解决国际网络访问问题。除此之外，Homebrew 和 npm 的默认服务器位于海外，在大陆网络下速度极慢甚至无法连接。建议配置国内镜像源：

Homebrew 镜像（macOS）：

bash

# 将以下三行添加到 shell 配置文件中（macOS 默认使用 zsh）：
echo 'export HOMEBREW_BREW_GIT_REMOTE="https://mirrors.ustc.edu.cn/brew.git"' >> ~/.zshrc
echo 'export HOMEBREW_CORE_GIT_REMOTE="https://mirrors.ustc.edu.cn/homebrew-core.git"' >> ~/.zshrc
echo 'export HOMEBREW_BOTTLE_DOMAIN="https://mirrors.ustc.edu.cn/homebrew-bottles"' >> ~/.zshrc

# 让配置立即生效：
source ~/.zshrc

上述命令的作用是告诉 Homebrew：不要去海外的默认服务器下载软件，改为去中国科学技术大学的镜像服务器下载，内容完全一致但速度快得多。

npm 镜像：

bash

# 将 npm 的下载源切换为淘宝镜像：
npm config set registry https://registry.npmmirror.com

# 验证是否生效：
npm config get registry
# 应输出：https://registry.npmmirror.com

提示：如果你使用推荐的原生安装器安装 Claude Code 本身，上述镜像配置不是必需的——原生安装器直接从 claude.ai 下载，不经过 Homebrew 或 npm。但当你后续需要安装其他工具（如 Node.js 包、Python 依赖等）时，这些镜像会帮你节省大量时间。

3 研究工作流的设计与搭建

本节介绍如何通过配置文件系统地管理 Claude Code 的行为。这是本文档的核心内容——如果说安装 Claude Code 是雇了一个助手，那么本节要做的就是为这个助手设计一套完整的工作制度。

3.1 前置知识：全局配置 vs 项目配置

在深入四层配置体系之前，你需要先理解一个基础概念：Claude Code 的所有配置文件都存在两个层级。

层级 位置 生效范围 是否共享
全局（用户级） ~/.claude/ 你所有项目共享 否，仅限你个人
项目级 你的项目/.claude/ 仅对当前项目生效 是，可通过 Git 分享给协作者

两个层级的目录结构几乎完全对称：

~/.claude/                          # 全局配置（你的 home 目录下）
├── CLAUDE.md                       # 全局指令，适用于所有项目
├── settings.json                   # 全局设置
├── rules/                          # 全局规则
├── skills/                         # 全局技能
└── agents/                         # 全局审查员

你的项目/                            # 项目配置
├── CLAUDE.md                       # 项目专属指令（这个在项目根目录）
└── .claude/
    ├── CLAUDE.md                   # 也可以放这里，效果相同
    ├── settings.json               # 项目设置
    ├── settings.local.json         # 个人的项目设置（不提交到 Git）
    ├── rules/                      # 项目规则
    ├── skills/                     # 项目技能
    └── agents/                     # 项目审查员

当 Claude Code 在某个项目文件夹中启动时，它会同时读取两个层级的配置并合并。如果全局和项目存在冲突，项目级优先。

设计原则是：跨项目通用的东西放全局（比如你个人的写作风格偏好、通用的 commit 规范），项目专属的东西放项目级（比如特定的数据分析流程、特定的论文格式要求）。

给新手的建议：刚开始不用纠结全局和项目怎么分。先把所有配置都放在项目的 .claude/ 目录里。等你开第二个研究项目时，自然会发现哪些文件每次都在复制——那些就应该提到 ~/.claude/ 全局目录中。

3.1.1 关于 AGENTS.md

你可能在一些教程中看到过 AGENTS.md 这个文件名。它最初由 OpenAI 的 Codex CLI 推广，现在已成为 Linux 基金会管理的开放标准，被多种 AI 编程工具支持（Claude Code、Codex CLI、Gemini CLI 等）。

Claude Code 的处理逻辑是：如果一个目录下没有 CLAUDE.md，它会自动查找 AGENTS.md 作为替代。换句话说，AGENTS.md 和 CLAUDE.md 的功能是等价的，区别在于 AGENTS.md 具有跨工具兼容性——如果你的团队中有人使用 Claude Code、有人使用 Codex CLI，维护一份 AGENTS.md 就能让所有人共享同一套项目规则。

对于本文档的读者，只使用 CLAUDE.md 即可。如果你未来有跨工具协作的需求，再考虑迁移到 AGENTS.md，或让 CLAUDE.md 引用 AGENTS.md 的内容。

3.2 概念总览：四层配置体系

回顾前文"人机协作的三个阶段"，Claude Code 的配置系统正是 Harness Engineering（智能体框架工程）的具体实现。你不是在写一条提示词，而是在搭建一套让 AI 持续、稳定、高质量工作的基础设施。

这套体系由四层构成，每一层解决不同的问题：

层级 文件位置 作用 类比
CLAUDE.md 项目根目录 项目的"宪法"，核心原则与全局规则 公司章程
Rules .claude/rules/ 按文件类型自动加载的领域知识 各部门的操作规范
Skills .claude/skills/ 可复用的工作流，用斜杠命令调用 标准作业程序（SOP）
Agents .claude/agents/ 专业化的审查员，各司其职 质检部门的不同岗位

它们之间的关系是递进的：CLAUDE.md 告诉 Claude"你是谁、遵守什么原则"；Rules 告诉它"处理这类文件时要注意什么"；Skills 告诉它"执行这个任务时按什么流程走"；Agents 则是独立的审查角色，从不同维度检验工作质量。

在动手之前，你的项目文件夹结构应该大致如下：

my-research-project/
├── CLAUDE.md                  # 项目宪法
├── .claude/
│   ├── rules/                 # 规则文件
│   ├── skills/                # 技能文件
│   └── agents/                # 审查员配置
├── paper/                     # 论文 .tex 或 .docx 文件
├── scripts/                   # R/Python/Stata 分析脚本
├── data/
│   ├── raw/                   # 原始数据（不修改）
│   └── derived/               # 清洗后的数据
├── output/
│   ├── tables/                # 回归表等
│   └── figures/               # 图表
└── quality_reports/
    ├── plans/                 # 任务计划
    └── session_logs/          # 会话日志

提示：你不需要一次性搭好所有层。完全可以从 CLAUDE.md 开始，用一段时间后再逐步添加 Rules、Skills 和 Agents。

3.3 第一层：CLAUDE.md — 项目宪法

3.3.1 它是什么

CLAUDE.md 是放在项目根目录下的一个 Markdown 文件。Claude Code 每次启动时都会自动读取它，相当于你在每次对话开始前对 Claude 说的一段"背景介绍 + 行为准则"。

一个关键的设计约束：Claude 大约能可靠地遵循 100-150 条自定义指令，而系统提示已经占用了约 50 条。这意味着留给 CLAUDE.md 的有效指令空间大约是 100-150 条。因此，CLAUDE.md 应该是一份精简的"宪法"（建议控制在 120-200 行以内），只放核心原则和指针，详细内容应当放到 Rules 和 Skills 中。

3.3.2 应该写什么

一个面向学术研究的 CLAUDE.md 通常包含以下部分：

  1. 项目基本信息：项目名称、研究领域、所在机构
  2. 核心工作原则：如"先规划再执行"、"执行后验证"、"所有修正记录到 MEMORY.md"
  3. 项目目录结构说明：让 Claude 知道数据放在哪、脚本放在哪、论文放在哪
  4. 工具链声明：你使用哪些语言和工具（R、Python、Stata、LaTeX 等）
  5. 常用命令速查表：列出你定义的 Skills 及其用途
  6. 质量标准：代码风格、论文格式、表格图表的规范
  7. 当前项目状态：各部分的进度，方便 Claude 在新会话中快速恢复上下文

3.3.3 实操示例

以下是一个面向社会科学研究的 CLAUDE.md 模板。你需要根据自己的项目修改方括号中的内容：

# CLAUDE.MD — 学术研究工作流

**项目:** [你的项目名称，如：家庭共餐对儿童发展的影响]
**机构:** [你的机构名称]
**研究领域:** 社会学/社会科学

## 核心原则
1. **先规划再执行** — 非平凡任务必须先进入 plan mode，草拟计划并保存到 quality_reports/plans/
2. **执行后验证** — 代码必须运行通过，才能报告完成
3. **质量门槛** — 80 分可 commit，90 分可提交审查，低于 80 必须修复
4. **记录修正** — 所有修正和经验教训记录到 MEMORY.md，防止跨会话重复犯错
5. **单一真相源** — 每个产出只有一个权威版本，其他版本从它派生

## 项目结构
- `paper/` — 论文 .tex 或 .docx 文件
- `scripts/` — R/Python/Stata 分析脚本
- `data/raw/` — 原始数据（**绝对不修改**）
- `data/derived/` — 清洗后的数据
- `output/tables/` — 回归表
- `output/figures/` — 图表
- `quality_reports/plans/` — 任务计划
- `quality_reports/session_logs/` — 会话日志（记录决策原因）

## 工具链
- **R:** tidyverse, fixest, ggplot2（数据分析、可视化）
- **Python:** pandas, statsmodels（数据处理）
- **Stata:** 计量经济学分析
- **LaTeX:** xelatex + bibtex（论文写作）
- **文献管理:** Zotero + .bib 文件

## 常用命令
| 命令 | 用途 |
|------|------|
| `/data-analysis [描述]` | 端到端数据分析 |
| `/proofread [file]` | 语法和一致性检查 |
| `/review-paper [file]` | 论文审查 |
| `/commit` | 暂存、提交、推送 |

## 质量标准
- **代码:** 每个脚本顶部设置随机种子；注释解释"为什么"而非"做了什么"
- **表格:** 标准误放括号内，标注显著性星号，注明固定效应
- **图表:** 统一配色方案，最小 12pt 字体，直接标注而非图例
- **论文:** 引用交叉验证，数学符号全文一致

## 当前状态
| 组件 | 状态 | 备注 |
|------|------|------|
| 数据清洗 | 完成 | survey_data_clean.csv |
| 描述性统计 | 进行中 | 缺少分组对比 |
| 回归分析 | 未开始 | |
| 论文初稿 | 未开始 | |

3.3.4 使用要点

  • 保持精简：抵制把所有细节都塞进 CLAUDE.md 的冲动。细节放 Rules 和 Skills 里。
  • 持续更新"当前状态"：每次结束工作会话时，更新项目状态表，这样下次启动 Claude Code 时它能立刻知道进度。
  • MEMORY.md 配合使用：在项目根目录创建一个 MEMORY.md，专门记录 Claude 犯过的错误和你纠正后的经验。Claude 每次启动都会读取它，避免重复犯同一个错。

3.4 第二层：Rules — 自动加载的领域知识

3.4.1 它是什么

Rules 是放在 .claude/rules/ 目录下的 Markdown 文件。与 CLAUDE.md 的区别在于：CLAUDE.md 每次会话都全量加载，而 Rules 可以设置路径作用域（path-scoping）——即只在 Claude 处理特定类型的文件时才自动加载。

为什么这很重要？因为前面提到，Claude 的指令遵循能力有上限。如果你把 R 代码规范、LaTeX 写作规范、Stata 使用规则全部塞进 CLAUDE.md，很快就会超出 Claude 的有效处理能力。路径作用域的设计让你可以按需加载：Claude 编辑 .R 文件时只看到 R 的规范，编辑 .tex 文件时只看到 LaTeX 的规范，互不干扰。

3.4.2 两种规则类型

始终加载的规则（无 paths 声明）：行为类似 CLAUDE.md 的补充条款，每次会话都会被读取。适合放通用的工作流程规则，如"先规划再执行"的详细流程。

路径限定的规则（有 paths 声明）：只在 Claude 操作匹配路径的文件时才加载。需要在文件顶部用 YAML frontmatter 声明作用范围。

3.4.3 实操示例

以下是几个对社科研究者最实用的规则文件。在你的项目中创建 .claude/rules/ 目录，然后把这些文件放进去。

示例 1：先规划再执行（始终加载）

文件路径：.claude/rules/plan-first-workflow.md

# 先规划再执行工作流

## 何时触发
任何非平凡任务（涉及多文件、多步骤、或预计超过 30 分钟的工作）
必须先进入 plan mode。

## 流程
1. 分析任务范围，识别依赖
2. 草拟计划，包含：目标、方法、涉及文件、验证步骤
3. 将计划保存到 `quality_reports/plans/YYYY-MM-DD_描述.md`
4. 呈现给用户，等待批准
5. 批准后按计划逐步执行
6. 完成后更新会话日志

## 为什么保存到磁盘
Claude Code 的上下文窗口会自动压缩。如果计划只存在对话中，
压缩后可能丢失。保存到磁盘后，即使切换会话也能恢复上下文。

示例 2：R 代码规范（路径限定）

文件路径：.claude/rules/r-coding-standards.md

---
paths:
  - "scripts/**/*.R"
  - "scripts/**/*.r"
---

# R 代码规范

## 基本要求
- 脚本顶部设置随机种子：`set.seed(20260401)` 格式为 YYYYMMDD
- 使用 tidyverse 风格：管道操作符 `|>` 而非 `%>%`
- 变量命名使用 snake_case
- 每个脚本顶部注明用途、输入数据、输出文件

## 统计分析
- 回归结果使用 `fixest` 包
- 报告异质性稳健标准误（HC1 或聚类标准误）
- 输出表格使用 `modelsummary` 或 `stargazer`

## 输出规范
- 表格保存到 `output/tables/`，格式为 .tex 或 .csv
- 图表保存到 `output/figures/`，格式为 .pdf 或 .png
- 图表使用 `theme_minimal()`，字体不小于 12pt

注意文件顶部的 paths: 声明——这告诉 Claude Code：只有在处理 scripts/ 目录下的 .R 文件时，才加载这条规则。

示例 3：LaTeX 论文写作规范（路径限定）

文件路径：.claude/rules/latex-writing.md

---
paths:
  - "paper/**/*.tex"
---

# LaTeX 论文写作规范

## 结构
- 主文件为 `paper/main.tex`
- 各章节拆分为独立 .tex 文件，用 `\input{}` 引入
- 参考文献使用 `natbib` + `\citet{}` / `\citep{}`

## 格式
- 表格使用 `booktabs` 包（\toprule, \midrule, \bottomrule）
- 所有浮动体（表格、图片）必须有 \label{} 和 \caption{}
- 交叉引用使用 `\autoref{}`

## 编译
- 编译命令：xelatex → bibtex → xelatex → xelatex
- 编译前检查是否有未定义的引用或标签

示例 4：Stata 代码规范（路径限定）

文件路径：.claude/rules/stata-standards.md

---
paths:
  - "scripts/**/*.do"
  - "scripts/**/*.ado"
---

# Stata 代码规范

## 基本要求
- 脚本开头：`clear all`、`set more off`、`set seed YYYYMMDD`
- 使用相对路径，不使用绝对路径
- 注释说明每个分析步骤的目的

## 回归输出
- 使用 `eststo` / `esttab` 或 `outreg2` 输出表格
- 报告聚类标准误时注明聚类变量
- 表格输出到 `output/tables/`

## 数据处理
- 原始数据从 `data/raw/` 读取，**永远不修改原始数据**
- 处理后的数据保存到 `data/derived/`
- 每次 merge 后检查 `_merge` 变量并记录匹配率

示例 5：会话日志规则（始终加载）

文件路径：.claude/rules/session-logging.md

# 会话日志规则

## 目的
Git 记录了你做了什么（what），会话日志记录你为什么这么做（why）。

## 三个写入时机

### 1. 计划批准后
创建 `quality_reports/session_logs/YYYY-MM-DD_描述.md`，记录：
- 目标和计划摘要
- 选择该方法的原因

### 2. 执行过程中
每当做出设计决策、发现问题、或偏离计划时，追加 1-3 行记录。
这是最重要的行为——上下文压缩会丢弃对话中的决策理由。

### 3. 任务完成时
追加最终状态、遗留问题、下次会话的建议起点。

3.4.4 使用要点

  • 路径作用域是关键优势：合理使用 paths: 可以让每种文件类型只加载与之相关的规则，避免浪费指令预算。
  • 规则文件不是越多越好：起步阶段 5-8 个规则文件已经足够。随着使用经验的积累再逐步添加。
  • 规则内容要具体：避免写"写出高质量的代码"这种笼统指令。要写"回归结果使用 fixest 包，报告异质性稳健标准误"这样 Claude 能直接执行的具体要求。

3.5 第三层：Skills — 可复用的工作流

3.5.1 它是什么

Skills（技能）是放在 .claude/skills/ 目录下的工作流模板。每个 Skill 是一个文件夹，里面包含一个 SKILL.md 文件以及可选的辅助资源（脚本、模板、参考文档等）。

Skill 与 Rule 的核心区别在于触发方式：Rule 是根据你操作的文件类型自动加载的背景知识，你感觉不到它的存在；Skill 则是一套显式的操作流程，你可以用斜杠命令手动调用（如输入 /data-analysis），也可以让 Claude 根据任务描述自动选择合适的 Skill。

用之前的组织制度类比来说：Rule 是每个部门默默遵守的操作规范，Skill 则是针对具体任务的标准作业程序（SOP）——当你说"现在要做数据分析"时，Claude 会拿出对应的 SOP 按步骤执行。

3.5.2 文件结构

一个 Skill 的基本结构如下：

.claude/skills/
└── data-analysis/              # Skill 名称（同时也是斜杠命令名）
    ├── SKILL.md                # 核心文件：指令和元数据
    ├── references/             # 可选：参考文档
    │   └── regression-checklist.md
    └── scripts/                # 可选：辅助脚本
        └── table-formatter.R

核心是 SKILL.md 文件，它由两部分组成：顶部的 YAML frontmatter（元数据，控制 Skill 的行为方式）和下方的 Markdown 内容（Claude 执行任务时遵循的具体指令）。

3.5.3 YAML Frontmatter 字段说明

yaml

---
name: data-analysis              # 显示名称，也是 /斜杠命令 的名字
description: >                    # 描述——Claude 根据这段话决定是否自动调用
  端到端数据分析。当用户要求进行统计分析、
  回归建模、数据探索或生成统计表格时使用。
argument-hint: "[分析描述]"       # 输入 /data-analysis 后的提示文字
---

几个重要的可选字段：

字段 作用 适用场景
disable-model-invocation: true 禁止 Claude 自动调用，只能手动 / 触发 高风险操作（如删除文件、提交代码）
user-invocable: false 不在 / 菜单中显示 只供其他 Skill 或 Agent 内部调用的后台知识
context: fork 在隔离的子上下文中运行，不污染主对话 耗时较长的研究或审查任务
agent: Explore 指定用哪个 Agent 执行 只需要读取不需要修改文件的任务
allowed-tools: Bash(R*), Read, Write 该 Skill 激活时允许使用的工具 控制权限边界

description 字段是关键：它不是给人看的说明文字，而是给 Claude 看的"触发条件"。写法应该是"当用户要求做 X 时使用"，而不是"这个 Skill 可以做 X"。前者帮助 Claude 判断何时自动调用，后者只是一句描述。

3.5.4 实操示例

以下是几个面向社科研究的 Skill，你可以直接在项目中创建对应的文件夹和文件。

示例 1：数据分析（/data-analysis）

文件路径：.claude/skills/data-analysis/SKILL.md

---
name: data-analysis
description: >
  端到端数据分析。当用户要求进行描述性统计、回归分析、
  数据探索、生成统计表格或可视化时使用。
argument-hint: "[分析描述]"
---

# 数据分析流程

执行数据分析时，严格按照以下步骤进行：

## 1. 理解需求
- 确认分析目标（描述性统计 / 回归 / 假设检验 / 探索性分析）
- 确认使用的数据文件（从 `data/raw/` 或 `data/derived/` 读取）
- 确认输出格式（表格 / 图表 / 两者都要）

## 2. 数据检查
- 报告观测数量、变量数量、缺失值情况
- 检查关键变量的分布（均值、标准差、极端值）
- 如果发现数据问题，**先报告给用户，不要自行处理**

## 3. 执行分析
- 脚本保存到 `scripts/` 目录
- 脚本顶部设置随机种子（`set.seed(YYYYMMDD)` 格式）
- 每一步分析都附注释说明目的
- 回归模型使用 `fixest` 包，报告聚类标准误

## 4. 输出结果
- 表格保存到 `output/tables/`（.tex 或 .csv）
- 图表保存到 `output/figures/`（.pdf 或 .png）
- 图表使用 `theme_minimal()`，字体不小于 12pt

## 5. 验证
- 代码必须完整运行通过，不能有报错
- 对结果做基本的合理性检查（系数方向、显著性、样本量）
- 如果结果异常，主动向用户报告

## 常见陷阱
- 不要在 `data/raw/` 中修改任何文件
- 不要使用 `rm(list=ls())`——它会清除用户环境中的其他对象
- 合并数据后必须检查 `_merge` 变量（Stata）或 join 后的行数变化（R）

示例 2：论文校对（/proofread）

文件路径：.claude/skills/proofread/SKILL.md

---
name: proofread
description: >
  学术论文校对。当用户要求检查语法、拼写、学术写作风格、
  引用一致性或论文格式时使用。
argument-hint: "[文件路径]"
---

# 学术论文校对流程

对指定文件进行逐段审查，检查以下维度：

## 语言与语法
- 语法错误、拼写错误、标点使用
- 主谓一致、时态一致性
- 避免口语化表达和非正式用语
- 被动语态不必完全消除，但避免过度使用

## 学术写作风格
- 论点表述是否清晰、有逻辑
- 段落之间的过渡是否自然
- 是否存在不必要的重复
- 术语使用是否全文一致

## 引用与参考文献
- 每个论点是否有对应的引用支撑
- 引用格式是否一致（APA / Chicago / 期刊特定格式）
- 文内引用与参考文献列表是否一一对应
- 检查是否有"孤立引用"（参考文献列表中有但正文未引用的）

## 输出格式
生成校对报告，按严重程度分类：
- 🔴 **错误**：必须修改（语法错误、引用缺失）
- 🟡 **建议**：强烈建议修改（表述不清、逻辑跳跃）
- 🔵 **风格**：可以考虑修改（更好的措辞选择）

示例 3：论文审查（/review-paper）

文件路径：.claude/skills/review-paper/SKILL.md

---
name: review-paper
description: >
  对论文进行全面学术审查。当用户要求审查论文质量、
  检查研究方法、评估论证逻辑或模拟审稿意见时使用。
argument-hint: "[文件路径]"

---

# 学术论文审查

以匿名审稿人的视角对论文进行全面审查。

## 审查维度

### 1. 研究问题与贡献
- 研究问题是否清晰明确？
- 对现有文献的贡献是什么？是否充分论证了贡献的重要性？
- 与最相近的已有研究相比，创新点在哪里？

### 2. 理论框架
- 理论框架是否适用于该研究问题？
- 核心概念的操作化定义是否合理？
- 假设推导的逻辑链是否完整？

### 3. 研究方法
- 数据来源和样本选择是否合理？
- 识别策略能否支撑因果推断？如果声称因果关系，识别假设是否可信？
- 是否存在遗漏变量、选择偏误、反向因果等问题？
- 稳健性检验是否充分？

### 4. 结果与讨论
- 结果呈现是否清晰？表格和图表是否自洽？
- 对结果的解释是否过度推断？
- 局限性讨论是否诚实且充分？

### 5. 写作质量
- 行文是否清晰、有逻辑？
- 篇幅是否合理？是否有可以精简的部分？

## 输出格式
生成一份结构化的审查报告：
1. **总体评价**（2-3 句话概括）
2. **主要优点**（3-5 条）
3. **主要问题**（按严重程度排列）
4. **具体修改建议**（对应到论文的具体段落或页码）
5. **小问题**（格式、拼写等）

文件路径：.claude/skills/commit/SKILL.md

---
name: commit

  暂存、提交并推送代码变更。当用户说"提交"、"commit"、

---

# Git 提交流程

## 步骤
1. 运行 `git status` 和 `git diff --stat` 查看变更概览
2. 将变更摘要呈现给用户确认
3. 用户确认后，执行：
   - `git add -A`
   - `git commit -m "简洁描述变更内容"`
   - 如果远程仓库已配置，询问用户是否 `git push`

## Commit Message 规范
- 中文撰写，简洁明了
- 格式：`[类型] 描述`
- 类型包括：`[数据]`、`[分析]`、`[论文]`、`[配置]`、`[修复]`
- 示例：`[分析] 添加基准回归模型，包含OLS和固定效应`

## 安全检查
- 提交前确认 `.gitignore` 已排除：原始数据文件、API 密钥、临时文件
- 如果发现敏感文件即将被提交，**立即警告用户并停止**

注意这个 Skill 设置了 disable-model-invocation: true——这意味着 Claude 不会自动触发提交操作，必须由你手动输入 /commit 才会执行。对于有副作用的操作（修改 Git 历史），这种显式控制是必要的。

3.5.5 进阶：Skill 中的动态上下文

Skill 支持在 Markdown 内容中嵌入 Shell 命令，Claude 调用 Skill 时会先执行这些命令，将结果注入上下文。语法是 ! 加反引号：

## 当前项目状态
- Git 状态: !`git status --short`
- 最近 5 次提交: !`git log --oneline -5`
- 数据文件列表: !`ls data/raw/`

这意味着每次调用 Skill 时，Claude 看到的不是静态文本，而是实时的项目状态。对于数据分析这类 Skill，你可以让它在启动时先自动获取数据文件列表、当前分支等信息。

3.5.6 使用要点

  • Skill 是文件夹，不是单个文件：即使目前只有 SKILL.md，也要创建文件夹结构（如 .claude/skills/data-analysis/SKILL.md）。后续你可以往文件夹中添加 references/（参考文档）和 scripts/（辅助脚本），实现渐进式信息披露。
  • description 字段决定自动触发：写得像触发条件（"当用户要求 X 时使用"），而不是功能说明。
  • 不要把 Skill 写成死板的步骤表：给出目标和约束，让 Claude 有灵活执行的空间。过于详细的步骤反而会限制 Claude 的判断力。
  • 起步建议 3-5 个 Skill：/data-analysis、/proofread、/review-paper、/commit 是社科研究者最常用的起点。随着你的使用经验积累，再逐步添加。
  • Gotchas 部分很有价值：在每个 Skill 末尾添加"常见陷阱"或"注意事项"，记录 Claude 曾经犯过的错误。这些是最高信号密度的内容——它们直接防止重复犯错。

3.6 第四层：Agents — 专业化审查员

3.6.1 它是什么

Agents（子代理 /审查员）是放在 .claude/agents/ 目录下的 Markdown 文件。每个 Agent 定义了一个独立的专业角色，拥有自己的指令、工具权限和行为模式。

Agent 与 Skill 的区别在于定位不同：Skill 是“做事的流程”——你调用 /data-analysis，Claude 按照流程执行分析；Agent 是“独立的人格角色”——你让 proofreader 审查论文时，它会以一个专业校对员的身份和视角来工作，而不是以Claude 的通用身份顺便帮你看看。

为什么需要独立的角色？因为 Claude 有一个常见的失败模式：对自己的工作过于宽容。如果你让 Claude 写完一段代码然后自己审查，它倾向于说“看起来不错”。但如果你让一个专门的 reviewer Agent 来审查——这个 Agent 被明确告知“你的职责是找问题，找不出问题说明你没认真看”——审查质量会显著提高。Sant'Anna 教授的工作流中，“批评者只读不改、修复者不能自我批准”的对抗式设计，正是利用了这个原理。

3.6.2 文件结构

Agent 的定义比 Skill 简单——只需要一个 Markdown 文件：

.claude/agents/
├── proofreader.md          # 语法校对员
├── domain-reviewer.md      # 领域审查员
└── r-reviewer.md           # R 代码审查员

每个文件同样由 YAML frontmatter（控制 Agent 的行为）和 Markdown 内容（Agent 的指令和角色描述）组成。

3.6.3 YAML Frontmatter 字段说明

---
name: proofreader
description: 学术论文语法和风格校对专家
model: sonnet                    # 使用的模型（sonnet 更快更便宜，opus 更强）
tools: Read, Grep, Glob         # 允许使用的工具
disallowedTools: Write, Edit    # 禁止使用的工具
permissionMode: plan            # 权限模式（plan = 只读 + 建议）
maxTurns: 50                    # 最大执行轮次
---

几个关键字段的解释：

字段 作用 设计意图
model 指定模型。可选 haiku（最快最便宜）、sonnet（平衡）、opus（最强）、inherit（与主会话一致） 简单审查任务用 sonnet 省钱，复杂推理用 opus
tools 允许使用的工具白名单 审查员通常只需要读取文件的权限
disallowedTools 禁止使用的工具 关键设计：让审查员无法修改文件，只能报告问题
permissionMode plan 表示只读建议模式，acceptEdits 表示可以修改文件 审查员用 plan，修复者用 acceptEdits
maxTurns Agent 执行的最大轮次，超过自动停止 防止 Agent 陷入无限循环
skills 预加载到 Agent 中的 Skill 列表 让 Agent 在执行时能调用特定的 Skill

3.6.4 Agent 与 Skill 如何协作

一个典型的工作模式是：Skill 定义工作流程，在流程中的某个环节调用 Agent 进行审查。例如：

  1. 你输入 /review-paper paper/main.tex
  2. Skill 启动论文审查流程
  3. 流程中调用 domain-reviewer Agent 审查研究方法
  4. 再调用 proofreader Agent 审查语言质量
  5. 两个 Agent 各自生成独立的审查报告
  6. Skill 汇总报告呈现给你

你也可以直接在对话中调用 Agent，而不通过 Skill：

> 用 proofreader 审查 paper/introduction.tex

Claude Code 会自动识别并调用对应的 Agent。

以下是三个对社科研究者最实用的 Agent。

示例 1：语法校对员（proofreader）

文件路径：.claude/agents/proofreader.md

---
name: proofreader
description: 学术论文语法和风格校对专家。审查语法、拼写、学术写作规范和引用一致性。
model: sonnet
tools: Read, Grep, Glob, Bash(wc*), Bash(grep*)
disallowedTools: Write, Edit
permissionMode: plan
maxTurns: 30
---

# 角色

你是一位资深的学术论文校对编辑，拥有丰富的社会科学期刊审稿经验。
你的职责是找出论文中的每一个语言问题。如果你审查后没有发现任何问题，
说明你审查得不够仔细——请重新检查。

# 审查维度

## 语法与拼写
- 主谓一致、时态一致性
- 冠词使用（a/an/the）
- 介词搭配
- 标点符号（特别注意逗号拼接、分号误用）

## 学术写作规范
- 避免口语化表达（"a lot of" → "substantial"）
- 避免模糊表述（"some researchers" → 具体引用）
- 避免过度使用被动语态
- 段落首句应该概括该段主题

## 引用一致性
- 文内引用格式是否统一
- 每个事实性陈述是否有引用支撑
- 参考文献列表与文内引用是否一一对应

## 术语一致性
- 同一概念全文是否使用同一术语
- 缩写首次出现时是否有完整形式

# 输出格式

按文件位置（行号或段落编号）逐条列出问题，每条包含：
- 📍 位置
- ❌ 原文
- ✅ 建议修改
- 📝 修改理由

最后给出总体评价：语言质量评分（1-10）和最需要关注的前 3 个系统性问题。

示例 2：领域审查员（domain-reviewer）

文件路径：.claude/agents/domain-reviewer.md

---
name: domain-reviewer
description: >
  社会科学研究方法和实质内容审查专家。审查理论框架、
  研究设计、识别策略、统计方法和结果解释。
model: opus
tools: Read, Grep, Glob, Bash(grep*), Bash(find*)
disallowedTools: Write, Edit
permissionMode: plan
maxTurns: 50
---

# 角色

你是一位资深的社会科学研究者，担任顶级期刊的匿名审稿人。
你的审查应当严格但建设性——指出问题的同时给出改进方向。
不要对作者客气，但也不要刻薄。

# 审查框架

## 1. 理论框架（权重：20%）
- 理论选择是否适合研究问题？
- 核心概念的定义是否清晰？操作化是否合理？
- 理论推导是否有逻辑跳跃？
- 与既有理论的对话是否充分？

## 2. 研究设计（权重：30%）
- 数据来源是否可靠？样本是否具有代表性？
- 变量测量是否有效度和信度？
- 识别策略是否能支撑因果推断的声称？
- 潜在的威胁：遗漏变量偏误、选择偏误、反向因果、测量误差

## 3. 统计方法（权重：25%）
- 所用方法是否适合数据结构和研究问题？
- 标准误的计算是否正确（聚类层级是否合理）？
- 稳健性检验是否充分且有说服力？
- 是否存在多重比较问题？

## 4. 结果解释（权重：15%）
- 对结果的解释是否超出了数据支撑的范围？
- 效应量的实际意义是否讨论了？
- 与既有文献的一致性 / 不一致性是否解释了？

3.6.6 进阶：对抗式审查模式（Critic-Fixer 循环）

前面提到 Sant'Anna 教授工作流中最强大的设计是对抗式审查。其核心思路是：

  1. Critic（批评者）：只有读取权限，不能修改文件。它的唯一职责是找问题。因为它无法自己修复，所以没有动机淡化问题的严重性。
  2. Fixer（修复者）：有修改权限，负责根据 Critic 的报告修复问题。但它不能自我批准——修复后必须再次提交给 Critic 审查。
  3. 循环：Critic 审查 → Fixer 修复 → Critic 再审查，最多 5 轮。如果 5 轮后仍有问题，报告给用户人工介入。

这种设计的效果是：避免了 Claude “对自己的工作说看起来不错”的常见失败模式。你可以在 Skill 中编排这种循环：

# 在某个 Skill 的流程中
## 审查-修复循环
1. 调用 domain-reviewer Agent 审查论文
2. 如果发现"致命"或"重大"问题：
   a. 根据审查报告修复问题
   b. 再次调用 domain-reviewer 审查修复后的版本
   c. 重复直到无重大问题，或达到 5 轮上限
3. 调用 proofreader Agent 审查语言
4. 汇总所有审查结果

对于刚起步的研究者，不需要立刻实现完整的 Critic-Fixer 循环。先从手动调用单个 Agent 开始——比如写完一段分析后，手动输入"用 r-reviewer 审查 scripts/regression.R"。等你对 Agent 的审查质量建立了信任，再考虑将其编排进自动化的 Skill 流程中。

3.6.7 使用要点

  • 审查员必须是只读的：通过 disallowedTools: Write, Edit 和 permissionMode: plan 确保审查员只能读取和报告，不能修改文件。这是保证审查独立性的关键设计。
  • 为不同任务选择不同模型：语法校对用 sonnet（快且便宜），领域审查用 opus（推理能力强）。不必所有 Agent 都用最贵的模型。
  • 起步 2-3 个 Agent 就够了：proofreader + domain-reviewer 是最实用的组合。等你用熟了再添加 r-reviewer、stata-reviewer 等专业审查员。
  • 持续积累 Gotchas：每次 Agent 漏掉了一个问题，或者给出了错误的建议，把这个情况添加到 Agent 的指令中。Agent 的质量是在使用中逐步打磨出来的。
  • 不要过度设计：Sant'Anna 教授有 10 个 Agent 和 22 个 Skill，但那是经过 6 轮以上迭代的成熟系统。你从 2-3 个 Agent 和 3-5 个 Skill 开始，完全足够。

3.7 项目管理实践：Git 版本控制与 MEMORY.md

前面四层配置体系解决了"怎么让 Claude 好好干活"的问题，但还有两个实际问题没有解决：Claude 改坏了东西怎么办？下次开新会话它又忘了上次犯的错怎么办？这就需要 Git 和 MEMORY.md。

3.7.1 为什么需要 Git

Claude Code 会直接修改你硬盘上的文件。这意味着它写错了一行代码、删错了一段论文，变更是真实发生的。没有 Git 的话，你可能根本不知道它改了什么，也没有办法回退。

Git 在这个工作流中承担三个角色：

  1. 安全网：改坏了可以回退（git checkout），改之前可以对比（git diff）
  2. 审查机制：每次让 Claude 完成任务后，先看它改了什么再决定是否保留
  3. 历史记录：记录了项目的每一步变更（what），配合会话日志（why）形成完整的决策链

3.7.2 Git 基础操作：从零开始

如果你从未用过 Git，以下是完整的操作流程。Git 已经预装在大多数 macOS 系统上；Windows 用户如果按照前文安装了 Git for Windows，也已经可以使用。

第一步：初始化

在终端中进入你的项目文件夹，初始化 Git 仓库：

bash

cd ~/Project/my-research-project

# 初始化 Git 仓库
git init

# 配置你的身份信息（只需做一次）
git config user.name "你的名字"
git config user.email "你的邮箱"

git init 会在文件夹中创建一个隐藏的 .git/ 目录，用于存储所有版本历史。这个目录不要手动修改或删除。

第二步：创建 .gitignore

在项目根目录创建一个 .gitignore 文件，告诉 Git 哪些文件不需要追踪。以下是一个适合社科研究项目的模板：

# 原始数据（通常太大或有隐私限制，不适合放进 Git）
data/raw/*.csv
data/raw/*.dta
data/raw/*.sav
data/raw/*.xlsx

# 但保留 raw 目录本身和数据说明文档
!data/raw/README.md
!data/raw/codebook/

# 编译产物
*.aux
*.log
*.bbl
*.blg
*.fls
*.fdb_latexmk
*.synctex.gz

# 系统文件
.DS_Store
Thumbs.db

# R 临时文件
.Rhistory
.RData
.Rproj.user/

# Python 临时文件
__pycache__/
*.pyc
.ipynb_checkpoints/

# API 密钥和敏感信息
.env
*.key

# 大型输出文件（可选，看你是否想追踪图表的版本变更）
# output/figures/*.pdf
# output/figures/*.png

提示：.gitignore 中的 ! 前缀表示"例外"——即使上面的规则排除了某个目录的文件，带 ! 的文件仍然会被追踪。

你可以用文本编辑器手动创建这个文件，也可以在 Claude Code 中直接说"帮我创建一个适合社科研究项目的 .gitignore"。

第三步：首次提交

创建好 .gitignore 后，把项目的初始状态保存为第一个版本：

bash

# 查看当前有哪些文件
git status

# 把所有文件加入暂存区
git add -A

# 创建第一个提交
git commit -m "[配置] 初始化项目结构和工作流配置"

三个命令的含义：git status 查看哪些文件发生了变化；git add -A 把所有变化"暂存"起来（相当于"准备提交"）；git commit -m "描述" 正式保存这个版本并附上说明。

第四步：日常工作流

之后的日常使用中，你和 Claude Code 的协作循环是：

1. 给 Claude 布置任务
   → Claude 修改文件

2. 查看 Claude 做了什么
   git diff                    # 查看具体改了哪些内容
   git diff --stat             # 查看改了哪些文件（概览）

3. 满意 → 提交
   git add -A
   git commit -m "[分析] 添加基准回归模型"

   不满意 → 回退
   git checkout -- .           # 撤销所有未提交的修改
   # 或只撤销某个文件：
   git checkout -- scripts/regression.R

如果你配置了前面的 /commit Skill，第 3 步可以简化为在 Claude Code 中输入 /commit，它会自动帮你完成 git add → git commit → 可选 git push 的流程。

什么时候应该 commit？

新手最常见的困惑是"多久提交一次"。以下是社科研究项目中推荐的提交时机：

时机 示例 Commit Message
项目初始化完成 创建目录结构、配置文件 [配置] 初始化项目结构和工作流配置
完成一个数据处理步骤 清洗完一个数据集 [数据] 清洗 CFPS 2018 家庭共餐变量
完成一个分析任务 跑完一组回归 [分析] 基准OLS回归和固定效应模型
完成一个写作段落/章节 写完引言初稿 [论文] 完成引言部分初稿
修复了一个问题 纠正了标准误的聚类层级 [修复] 标准误改为省级聚类
修改了配置文件 添加了新的 Rule 或 Skill [配置] 添加 R 代码规范规则
一天工作结束时 不管做到哪都先提交 [进度] 4月1日工作存档

核心原则是：每完成一个你能用一句话描述的独立任务，就提交一次。宁可多提交，不要攒一大堆变更一次提交——攒得越多，出了问题越难定位是哪一步改坏的。

不要在以下时候 commit：

  • Claude 还在执行任务的中途（等它做完再看结果）
  • 你还没有用 git diff 检查过变更内容（先审查再提交）
  • 代码还跑不通或有明显错误（先修好再提交，保持每个版本都是"能用的"）

常用 Git 命令速查

命令 作用 什么时候用
git status 查看当前有哪些变更 随时
git diff 查看具体改了什么 Claude 完成任务后、提交前
git add -A 暂存所有变更 准备提交时
git commit -m "描述" 提交并保存版本 一个阶段性任务完成时
git log --oneline -10 查看最近 10 次提交记录 回顾历史
git checkout -- . 撤销所有未提交的修改 Claude 改坏了想回退
git checkout -- 文件名 撤销某个文件的修改 只想回退一个文件

给完全没用过 Git 的人的建议：刚开始只需要记住三个命令的循环——git diff（看改了什么）→ git add -A（暂存）→ git commit -m "描述"（提交）。其他命令用到的时候再查。你直接问 Claude Code "帮我看看现在有什么变更"或"帮我提交一下"，它会帮你执行对应的 Git 命令。
